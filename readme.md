# Norde Maison — итоговая документация проекта (backend)

### Что это за проект

Это backend интернет‑магазина Norde Maison. Он хранит каталог и остатки, даёт корзину и оформление заказа, подключает оплату, отправляет уведомления, и предоставляет админ‑панель для управления товарами и заказами. Фронтенд — это интерфейс, backend — данные и бизнес‑правила.

### Технологии и почему они

- **Python + Django**: быстрый и надёжный фундамент. Django даёт ORM, миграции, встроенную админку и базовую безопасность. Для e‑commerce это удобно, потому что много сущностей и правил.
- **Django REST Framework (DRF)**: выбран как стандарт для построения REST API в Django‑экосистеме. Он даёт чёткую архитектуру API (views/serializers/permissions), централизованную валидацию входных данных и единый формат ответов. Это снижает количество ошибок на границе “фронт↔бэк” и упрощает сопровождение: контракт API становится прозрачным и предсказуемым.
- **PostgreSQL**: выбрана как надёжная транзакционная реляционная БД, где критичны целостность и согласованность данных. Для магазина важно, чтобы связи и ограничения работали строго: уникальность вариантов товара, корректная структура “заказ ↔ позиции”, невозможность создать дубли корзины, сохранение истории заказов. PostgreSQL хорошо подходит под такие задачи и стабильно работает под нагрузкой.
- **Token‑авторизация**: простой и понятный способ для SPA/клиентского приложения — логин → токен → запросы с токеном.
- **YooKassa**: выбран платёжный провайдер, потому что даёт промышленный платёжный контур и предсказуемую модель подтверждения оплаты: редирект на оплату, идентификатор платежа, уведомления о результате и проверка статуса. Для магазина это критично: оплата должна подтверждаться однозначно, а заказ — автоматически переходить в работу без ручной сверки.
- **Email + Telegram**: email — клиенту (официальные уведомления), Telegram — менеджерам (оперативные сообщения о заказах и статусах).
- **openpyxl (Excel)**: бизнес‑отчётность в понятном формате (продажи, средний чек, топ товаров, статистика по избранному).

### Как это работает по основному сценарию покупки

Пользователь открывает каталог: backend отдаёт категории/подкатегории и товары. У товаров есть варианты (цвет/размер) и остатки, и пользователь видит только то, что реально доступно к продаже.

Дальше пользователь выбирает конкретный вариант и добавляет в корзину. Корзина хранит именно вариант товара, а не “товар вообще”. Backend сразу контролирует ограничения: нельзя взять больше, чем есть на складе, и есть лимит на количество одного варианта.

Перед оформлением есть предпросмотр: backend собирает корзину и показывает итог. На оформлении пользователь выбирает страну и способ доставки (ПВЗ или курьер), а backend применяет настройки доставки по регионам.

Если выбрана оплата, backend создаёт платёж YooKassa и отдаёт ссылку на оплату. После успешной оплаты заказ автоматически подтверждается и переходит в работу.

### Что происходит после оплаты (коммуникации и статус‑менеджмент)

После оформления клиент получает email‑подтверждение. Менеджеры получают Telegram‑уведомление с составом заказа, данными доставки и ссылкой на заказ в админке.

Дальше менеджер работает в админке: переводит заказ по статусам “в сборке” → “в пути” → “доставлен” (или “отменён”). Система автоматически отправляет уведомления клиенту и в Telegram о смене статусов.

### Надёжность: склад и “не потерять оплату”

Две ключевые проблемы e‑commerce: не продать то, чего нет, и не потерять оплату.

- Остатки контролируются на стороне backend: нельзя превысить склад при добавлении в корзину и при оформлении.
- Подтверждение оплаты сделано устойчиво: есть уведомление от платёжной системы и дополнительные проверки статуса, чтобы заказ корректно переходил из “ожидает оплаты” в работу.
- Если оплату не сделали, заказ отменяется, а остатки возвращаются — чтобы склад не “залипал”.

### Честные ограничения и куда улучшать

Для production я бы вынес секреты (ключи/пароли) в переменные окружения и секрет‑хранилище. И фоновые операции (уведомления/проверки) вынес бы в очередь задач (Celery/RQ), чтобы это масштабировалось и не зависело от процесса веб‑сервера.

## 1) Описание “для заказчика” (коротко и понятно)

Backend Norde Maison — это “мозг магазина”: он держит данные о товарах и остатках, оформляет заказы, интегрируется с оплатой YooKassa, автоматически уведомляет клиента и менеджеров, и даёт админ‑панель для управления магазином и выгрузки отчётов в Excel.

Если по‑простому: **витрина → корзина → заказ → оплата → уведомления → статусы → отчёты**.

---

## 2) Назначение проекта

Проект `norde_maison` — серверная часть интернет‑магазина на Django + Django REST Framework.

Сервер предоставляет:

- **REST API** для клиентского приложения (каталог, корзина, избранное, оформление заказа, профиль).
- **Админ‑панель Django** для управления товарами/вариантами/подкатегориями/заказами и генерации отчётов.
- **Интеграции**: оплата **YooKassa**, email‑уведомления, Telegram‑уведомления менеджерам.

---

## 3) Технологический стек

- **Python**: 3.12+ (см. `pyproject.toml`)
- **Backend**: Django 6, Django REST Framework
- **Auth**: DRF TokenAuthentication
- **DB**: PostgreSQL (`psycopg2-binary`)
- **Медиа**: Pillow
- **Админка и UX админки**: `adminsortable2`, `django-image-uploader-widget`, `django-tinymce`, `django-colorfield`
- **Отчёты**: Excel через `openpyxl`
- **Интеграции**: `yookassa`, `requests` (Telegram Bot API)
- **CORS**: `django-cors-headers`

---

## 4) Структура проекта (Django apps)

- `norde_maison/` — настройки проекта и корневой роутинг API
- `users/` — регистрация/логин/профиль/подтверждение email/смена и сброс пароля
- `catalog/` — категории/подкатегории/товары/варианты/поиск/фильтры
- `cart/` — корзина пользователя
- `favorites/` — избранное пользователя + Excel‑отчёт в админке по популярности избранного
- `orders/` — оформление заказа/оплата/статусы/уведомления + Excel‑отчёты по продажам
- `shop_config/` — конфиг доставки и Telegram/ссылок (Singleton‑модели)

Инфраструктура:

- `templates/` — шаблоны email и страниц подтверждения, шаблоны для админки
- `static/` — статические файлы (в т.ч. для админки)
- `media/` — загружаемые изображения и отчёты (`media/reports/`)

---

## 5) Роли и доступ

По умолчанию API закрыт авторизацией, а публичные разделы открываются явно.

- **Гость**: просмотр каталога, регистрация/подтверждение email, сброс пароля, получение публичных настроек сайта.
- **Пользователь**: корзина, избранное, оформление заказа и просмотр истории/статусов.
- **Админ/менеджер**: управление каталогом и заказами через `/admin/`, создание отчётов Excel, настройка доставки/Telegram/ссылок.

---

## 6) Ключевые сущности данных (понятно, без деталей кода)

- **Категория/подкатегория**: структура каталога (в т.ч. разделение по полу).
- **Товар**: описание и цены (в нескольких валютах), главное фото и галерея.
- **Вариант товара**: цвет/размер/остаток — именно это продаётся и кладётся в корзину.
- **Корзина**: набор выбранных вариантов с количеством.
- **Избранное**: понравившиеся товары (для удобства клиента и аналитики).
- **Заказ**: контакты/доставка/итог, статус, привязка к платежу YooKassa.
- **Позиции заказа**: фиксируют состав покупки (в т.ч. “снимок” цены на момент покупки).
- **Отчёт**: файл Excel, сформированный админом за период.

---

## 7) Оплата и статусы заказа (логика)

Основные статусы: “ожидает оплаты” → “в сборке” → “в пути” → “доставлен” / “отменён”.

Оплата через YooKassa выбрана, потому что она закрывает ключевые требования магазина: надёжное подтверждение оплаты, понятный клиентский сценарий оплаты через редирект, возможность однозначно связать платёж с заказом и получать результат оплаты как по уведомлениям, так и через проверку статуса.

Оплата через YooKassa:

- при оформлении с оплатой формируется платёж и отдаётся ссылка на оплату;
- после успешной оплаты заказ автоматически переводится в работу;
- если оплата не прошла/истекло время ожидания — заказ отменяется, а остатки возвращаются.

---

## 8) Уведомления

- **Email**:
  - подтверждение регистрации
  - сброс пароля
  - подтверждение заказа
  - уведомления о смене статуса (в пути/доставлен/отменён)
- **Telegram** (для менеджеров):
  - новый заказ и его состав
  - уведомления об изменениях статуса
  - фиксация любых событий по заказу в операционном чате: создание заказа, подтверждение оплаты, смена статуса, отмена/таймаут, чтобы менеджеры видели всю историю действий и ничего не терялось в ручной коммуникации

---

## 9) Отчётность (Excel)

В админке доступны отчёты:

- **по продажам** (в т.ч. средний чек, топ‑товары, продажи по периоду)
- **по избранному** (популярные товары в избранном) с выгрузкой Excel
